# 読み上げ内容が ai-twitch-cast と違う — 差分の調査と整理

## 目的・背景

ai-monitor の `--mode server` 音声読み上げは、参考実装 `~/ai-twitch-cast` の「ちょビ」を移植して作った（`docs/plans/archive/claude-progress-voice.md`）。しかし実際に聞くと **読み上げの内容が ai-twitch-cast と体感で違う**。本タスクは「どこがどう違うか」を軸ごとに洗い出して整理し、次の改善（どの差を寄せ、どれは意図的に残すか）の判断材料にする。

移植時に「Twitch 配信系（アバター・リップシンク・授業・チャット投稿・2 人会話）を全部そぎ落とし、`event → ペルソナ短文 → TTS → 再生` の最小フロー」へ意図的に縮小している。差分の一部はこの設計判断の結果であり、一部は移植時の劣化。両者を切り分けるのが本調査のゴール。

**関連 TODO**: 「読み上げが途中で途切れる。文字数制限は AI でのテキスト生成で行い、読み上げは全てのテキストを読む」。本件の出力長キャップ（`PERSONA_MAX_CHARS=60`・末尾 `…` 切り詰め）と直結するので、整理結果は隣タスクの方針決定にも使う。

## 対応方針（調査の進め方）

両実装の「読み上げテキストが出来るまで」を同じ軸で並べて diff する。

調査軸:
1. 生成モデル（テキスト / TTS）
2. キャラ構成（独白 / 対話）
3. system プロンプト（層構成・人格の中身）
4. 投入コンテキスト（LLM に渡す素材の量と種類）
5. 出力の長さ・形式
6. 感情 / SE（演出メタデータ）
7. 繰り返し防止・バリエーション
8. トリガ / タイミング（いつ・何件で喋るか）
9. TTS 声 / スタイル
10. 言語切替
11. 検証可能性（プロンプト全文の可視化）

各差分を **「意図的にそぎ落とした（配信専用機能）」/「移植時に劣化した（寄せるべき）」/「要判断」** の 3 分類に振る。

## 主要な所見（なぜ内容が「薄く / 違う」と感じるか）

整理した結果、体感差の主因は次の順で効いている。

1. **④ コンテキスト枯渇が最大要因**。ローカルは「直近の assistant **テキスト** 1 件」しか LLM に渡していない。Claude が実際に *何をしたか*（編集したファイル・実行コマンド・検索・サブエージェント）はそもそも入力に無いため、ちょビは具体を語れず「終わったよ」「確認待ちだよ」程度の一般論に収束する。ai-twitch-cast は「直近 10 アクション」を渡すので作業の中身を実況できる。→ ここを `uplink.ts` の `detail` に「直近アクション列」を載せる方向に寄せるのが、内容差を埋める一番効くレバー（抽出元の素材は `transcript.ts` / `entry-source.ts` に既にある）。
2. **⑤ 出力長 + ② 独白**。1 文 50 字キャップは、2〜4 発話の掛け合いに比べて情報密度・尺ともに小さい。末尾 `…` 切りは「途切れる」の元（隣 TODO）。
3. **③ 人格書き換え + ⑦ 反復**。原文の配信者ちょビ（好奇心・ツッコミ）を「進捗実況アシスタント」に均した上、キャッシュで同入力=同文のため、味が薄く・繰り返しがちに感じる。Persona / Self-Note の記憶層も無い。
4. **⑥⑩ 演出（emotion/SE/言語切替）と ⑧ 途中実況の主従**は配信専用機能で、ローカルでそぎ落としたのは設計どおり。内容差の主因ではない。

## この後の判断ポイント（Phase 2 でユーザー確認）

整理だけでは決められない設計判断。改善着手前に確認する。

- **A. コンテキスト拡張**: `detail` を「直近 assistant text のみ」→「直近 N アクション（編集ファイル / コマンド / 検索）」に広げるか。広げる場合、redaction（`redaction.ts`）とプライバシー方針（ミラーで外部通過）に収まる粒度をどこに置くか。
- **B. 形式**: ちょビ 1 人独白のままか、2 人対話に寄せるか（声 Aoede 追加・尺増・コスト増を伴う）。
- **C. 長さ**: 50 字キャップを外し「文字数制限は生成側で・読み上げは全文」にするか（隣 TODO とまとめて対応）。
- **D. 反復回避**: 直近発話を prompt に入れる / キャッシュ方針を見直すか。
- **E. モデル**: テキスト生成を Haiku のままにするか、Gemini に合わせるか（口調再現性 / コスト）。

## 影響範囲（改善フェーズで触る想定。本調査では変更なし）

- `ai-monitor/src/uplink.ts` — `detail` に何を載せるか（A の核）。`VoiceEventDetector` の素材取得。
- `ai-monitor/src/persona.ts` — プロンプト構成（③）・出力上限（⑤）・キャッシュ（⑦）。
- `ai-monitor/voice-persona.json` — 人格 / ルール（③）。
- `ai-monitor/src/transcript.ts` / `entry-source.ts` — アクション列の抽出元（A の素材）。
- `ai-monitor/src/tts.ts` — 2 声化する場合（B）。
- `redaction.ts` — コンテキスト拡張時のマスク粒度（A）。

## テスト方針（調査の裏取り）

- **実プロンプトのダンプ**: 実セッションを 1 ターン回し、ローカルが実際に組み立てた `buildPersonaPrompt` の system / user と最終 text を 1 件ダンプして、本整理表の「現状」列が机上でなく実物と一致することを確認する（⑪ の不足を一時的に補う）。
- **既存テスト**: `persona.test.ts`（プロンプト純関数）・`voice-pipeline.test.ts` を確認し、改善時に変えるべきアサーションの所在を押さえる。
- ai-twitch-cast 側は読み取り専用調査（変更しない）。

## Phase / Step 分解

- **Phase 1: 差分の洗い出しと整理（本ファイル）** ✅ 完了
  - 両実装を 11 軸で diff し、3 分類（意図的 / 劣化 / 要判断）に整理。主因を特定。
- **Phase 2: 分類の裏取りと「寄せる / 寄せない」方針決定**
  - 実プロンプトを 1 件ダンプして「現状」列を実物照合。
  - 上記 A〜E をユーザーと確認し、各差分の対応方針を確定。
- **Phase 3: 改善タスク化**
  - 確定した方針を個別の実装タスク（別プラン / TODO）に落とす。隣 TODO（途切れ / 全文読み上げ）と統合可能なものはまとめる。

## 残課題・備考

- ai-twitch-cast は元々「作業中の途中実況」が主用途、本実装は「完了 / 承認待ち通知」が主用途。**用途差がそもそもある**点は、内容差を評価するときに常に念頭に置く（同じにするのが正解とは限らない）。
- ai-twitch-cast の設計思想ドキュメント `~/ai-twitch-cast/docs/speech-generation-flow.md` / `docs/character-prompt.md` は Phase 2 で人格再現方針を詰めるときの一次資料。
